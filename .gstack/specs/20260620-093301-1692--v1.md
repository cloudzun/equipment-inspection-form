# Spec: 车间设备巡检登记表 (v1)

## Context

车间目前使用纸质卡片登记设备巡检结果——2名巡检员每轮各填约20台设备，
管理者事后需要翻纸质卡片查历史。纸质卡容易丢失，历史查询耗时且不可靠。
本次用电子化方案替代，扫码即填，提交后自动集中存储，管理者可按设备/日期/人员筛选。

## Current State

- 无代码库，纯新构建
- 设计文档已通过 /office-hours + /autoplan 评审
- 架构决策：方案 A（单文件 HTML）+ 轻量 sync 端点（Express + SQLite WAL）

## Proposed Change

构建 2 个文件：

1. **index.html** — 巡检员表单页 + 管理者历史查询页，单文件自包含
2. **server.js** — Express sync 端点，接收 POST JSON 写入 SQLite

巡检流程：巡检员手机扫码 → 浏览器打开 `http://<server>/?device=<id>` → 选状态 → 填姓名备注 → 提交。提交后前端写 IndexedDB（本地缓存）并 POST 到 server.js。POST 成功 → 记录标记"已同步"。POST 失败 → 记录打 retry 标记，下次页面加载时重试。

管理者查历史：打开 `http://<server>/?view=history` → 按设备编号/日期/巡检人筛选。

### Architecture

```
┌──────────────┐     POST /api/inspections      ┌──────────────┐
│  index.html  │  ──────────────────────────>   │  server.js   │
│  (browser)   │  <── 201 + server_timestamp    │  (Express)   │
│              │                                 │              │
│  IndexedDB   │                                 │  SQLite WAL  │
│  (cache)     │                                 │  inspections │
└──────────────┘                                 └──────────────┘
```

### Data Schema

**前端 IndexedDB（Browser）:**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | auto-increment | 主键 |
| device_id | string | 设备编号 |
| status | enum | 正常 / 异常 / 待维修 |
| inspector | string | 巡检人姓名 |
| note | string | 备注 |
| client_created_at | ISO 8601 | 客户端提交时间 |
| synced | boolean | 是否已同步到服务器 |
| server_created_at | ISO 8601 | 服务器返回的时间戳（sync 成功后填充） |

**后端 SQLite（Server）:**

```sql
CREATE TABLE IF NOT EXISTS inspections (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  device_id TEXT NOT NULL,
  status TEXT NOT NULL CHECK(status IN ('正常', '异常', '待维修')),
  inspector TEXT NOT NULL,
  note TEXT DEFAULT '',
  client_created_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  submitter_ip TEXT DEFAULT ''
);
CREATE INDEX idx_inspections_device ON inspections(device_id);
CREATE INDEX idx_inspections_created ON inspections(created_at);
CREATE INDEX idx_inspections_inspector ON inspections(inspector);
```

### API Contract

**POST /api/inspections**

Request:
```json
{
  "device_id": "shadds-01",
  "status": "正常",
  "inspector": "张三",
  "note": "运行平稳",
  "client_created_at": "2026-06-20T10:30:00.000+08:00"
}
```

Response (201):
```json
{
  "id": 42,
  "created_at": "2026-06-20T10:30:05.123+08:00"
}
```

Response (400 - validation failure):
```json
{
  "error": "invalid_field",
  "field": "device_id",
  "message": "设备编号不在设备列表中"
}
```

Response (500 - server error):
```json
{
  "error": "internal_error",
  "message": "服务器内部错误，请重试"
}
```

**GET /api/inspections**

Query params:
- `device_id` (optional): 按设备编号筛选
- `date_from` (optional): 起始日期 ISO 8601
- `date_to` (optional): 结束日期 ISO 8601
- `inspector` (optional): 按巡检人筛选
- `limit` (default 200): 最大返回条数

Response (200):
```json
{
  "records": [
    {
      "id": 42,
      "device_id": "shadds-01",
      "status": "正常",
      "inspector": "张三",
      "note": "运行平稳",
      "created_at": "2026-06-20T10:30:05.123+08:00"
    }
  ],
  "total": 1
}
```

### Form Validation

| 规则 | 字段 | 行为 |
|------|------|------|
| 必填 | device_id | 下拉列表选择，不选不能提交 |
| 必填 | status | 三个状态选一 |
| 必填 | inspector | 不能为空或纯空格，trim() 后校验 |
| 可选 | note | 最大 1000 字符 |
| 防重复 | 整个表单 | 提交后按钮 disabled + 显示 "已提交 ✅" 2 秒后清空 |

### Security

- **XSS**: 历史查询页渲染记录时使用 `textContent`，不使用 `innerHTML`。显式转义 `<` `>` `&` `"` `'`
- **CORS**: `Access-Control-Allow-Origin` 设为前端 origin，不放 `*`
- **设备白名单**: HTML 中嵌入 DEVICE_LIST 数组 + 服务器端也校验
- **无认证**（有意识的设计选择）: 内网信任边界。服务器记录 `submitter_ip` 作为基础审计
- **Content-Type**: 服务器只接受 `Content-Type: application/json`

### QR Code Format

每台设备的二维码编码为完整 URL：
```
http://<server-ip>:3000/?device=shadds-01
```
二维码生成工具：`qrencode` 或在线生成器。共需要 20 个二维码。

### Retry Mechanism

1. 提交 → 写 IndexedDB（`synced: false`）
2. POST /api/inspections
3. 成功 → 更新 IndexedDB（`synced: true`, `server_created_at`）+ 显示 "已提交 ✅"
4. 失败 → 显示 "已保存，等待同步 ⏳" + 记录 retry 标记
5. 页面加载时：遍历 IndexedDB 中所有 `synced: false` 记录，逐一 POST
6. 对旧记录：去重（`client_created_at` + `device_id` + `inspector` 三字段相同的跳过）

### Acceptance Criteria

1. 巡检员扫描设备二维码，浏览器自动打开到该设备的登记页，设备编号已预填
2. 巡检员手动输入/选设备编号也能登记（QR 码花了的情况备选入口）
3. 提交一次巡检记录 ≤30 秒（从扫码到看到 "已提交 ✅"）
4. 提交显示绿色确认动画，2 秒后表单清空可填下一台
5. 服务器离线时，记录存在浏览器中，服务器恢复后自动同步
6. 管理者打开历史页面，输入设备编号，3 秒内看到该设备最近 50 次巡检记录
7. 管理者可按设备编号、日期范围、巡检人任意组合筛选
8. 两名巡检员同时提交记录，服务器不丢数据不损坏
9. 巡检人中文姓名和备注往返提交不出现乱码
10. 所有设备编号只能从下拉列表选，不能手输自由文本
11. 服务器文件在 100 并发 POST 场景下不丢记录

### Deployment

1. 服务器安装 Node.js（已有）
2. `npm install express better-sqlite3`
3. `node server.js`（或用 pm2 管理进程持久化）
4. 确保 `index.html` 和 `server.js` 同目录（Express 托管静态文件）
5. 服务器绑定静态 IP 或内网 DNS 名
6. 生成 20 个设备二维码贴上对应设备

### Out of Scope

- 拍照附件上传
- 离线模式（无网络也能填，有网再同步）—— 当前是"网络瞬断重试"，不是离线
- 多车间/多班组权限管理
- 推送通知
- 检修工单系统
- 巡检完成率统计面板
- 登录/认证系统（内网信任边界）
- 打印功能

### Rollback Strategy

- **代码回滚**: `git revert <commit>` + 重新 `node server.js` 重启即可。无数据库迁移，无 breaking change
- **数据恢复**: SQLite 单文件 `inspections.db`。部署前备份，有问题直接替换旧文件
- **渐进上线**: 先让一名巡检员试一天，确认没问题后第二个巡检员加入。纸质表格保留并行运行一周，确认电子版正常后废弃
