## Problem Statement

车间目前使用纸质表格登记设备巡检结果。纸质表格容易丢失、查找历史记录麻烦——巡检员每次巡检完要填设备状态，管理者需要能按设备、日期、人员查看历史。目标是用电子化方案替代纸质表格，保留简单性，大幅提升记录可靠性和查询效率。

## What Makes This Cool

- **零安装**：手机浏览器打开就能填，不需要装 App、不需要注册账号
- **扫码即填**：每台设备贴个二维码，巡检员扫一下直接跳转到该设备的登记页——比翻纸省事太多
- **即时历史查询**：纸的时候翻半天，现在按设备/日期/人员几秒筛出来
- **极简架构**：一个 HTML 文件搞定所有，部署就是放到服务器上，比搭建前后端快 10 倍

## Constraints

- 车间有网络覆盖（WiFi/4G 都可以）
- 巡检区域不需要离线功能
- 异常处理走即时通讯（电话/微信），不依赖系统推送通知
- 设备数量级别：中小规模车间，几十到几百台设备
- 无需多车间多班组权限——巡检员填，管理者查

## Premises

1. 纸质表格是真正的痛点——频繁发生的实际问题，不是偶尔的不便。车间人员愿意用手机扫码代替纸笔。
2. 扫码是最自然的入口——每台设备贴二维码，扫了直接到登记页。
3. 不需要离线功能或即时推送通知——有网络覆盖，异常情况巡检员会当场联系维修。

## Approaches Considered

### Approach A: 单文件 HTML（最小可行）

- **Summary**: 一个自包含的 HTML 文件，内嵌表单、IndexedDB 存储、历史查询。部署就是放到任意 Web 服务器上（或直接用 npx serve）。
- **Effort**: S（Human: ~半天 / CC: ~30min）
- **Risk**: Low — 无外部依赖，不依赖数据库安装，任何浏览器都能跑
- **Pros**:
  - ✅ 无需后端、无需数据库、无需构建工具——一个文件直接能用
  - ✅ 数据存在浏览器 IndexedDB，无需服务器端存储，降低运维成本
  - ✅ 可通过 URL 参数 `?device=设备编号` 实现扫码直达——每台设备的二维码编码为完整 URL：`http://<服务器IP>/index.html?device=<设备编号>`，扫码后浏览器直接打开该设备的登记页
  - ✅ 部署只需放到任何静态文件服务器上（nginx/caddy/甚至 GitHub Pages）
- **Cons**:
  - ❌ 数据分散在每个巡检员的手机浏览器里，管理者需要在一台电脑上汇总查看
  - ❌ 换台手机就看不到之前的数据（除非导出/备份）
  - ❌ 无法多人协作——每个人看到的是自己的记录
- **Reuses**: 零依赖，纯 HTML/CSS/JS

### Approach B: 前后端分离（标准架构）

- **Summary**: 前端 HTML/React 表单 + 后端 SQLite 数据库。巡检员手机浏览器填表，数据集中存服务器。管理者通过 Web 页面查历史。支持多设备多巡检员。
- **Effort**: M（Human: ~1天 / CC: 1-2h）
- **Risk**: Medium — 需要部署后端服务，涉及 CORS、数据库管理等
- **Pros**:
  - ✅ 数据集中存储，管理者实时看到所有巡检记录
  - ✅ 多巡检员可同时使用，数据不分散
  - ✅ SQLite 单文件数据库，无需安装数据库服务，迁移/备份简单
  - ✅ 可后续扩展：附件上传、统计报表、用户登录
- **Cons**:
  - ❌ 需要部署后端服务（Node.js/Python），增加运维复杂度
  - ❌ 需要处理网络故障、并发等边界情况
  - ❌ 比单文件方案多几倍代码量和调试时间
- **Reuses**: 单文件方案的表单 UI 可直接迁移到前端

### Approach C: 完整版（理想架构）

- **Summary**: React 前端 + SQLite/PostgreSQL + 扫码自动识别设备 + 拍照附件 + 多维历史查询 + 巡检完成率统计看板。
- **Effort**: L（Human: ~2-3天 / CC: 3-4h）
- **Risk**: Medium-High — 功能多、状态管理复杂、图片存储需额外处理
- **Pros**:
  - ✅ 功能完整：扫码、拍照、统计、多设备多用户
  - ✅ 长期扩展性好——未来加维修工单、保养提醒很容易
  - ✅ 专业感强，车间管理者可以看到仪表盘
- **Cons**:
  - ❌ 开发量大，很多功能当前阶段用不上
  - ❌ 维护成本高——上线后发现真正需要的可能只是核心的一部分
  - ❌ 与"尽快替代纸质表格"的目标相比，范围过大
- **Reuses**: 方案 B 的前后端可作为基础

## Recommended Approach

**Approach A + Lightweight Sync（修订后）**

理由：目标是用最快速度干掉纸质表格。方案 A 半天就能让车间实际用起来。在此基础上增加一个轻量 sync 端点（~20 行 Node.js/Express，每次提交自动 POST 到服务器 SQLite 文件），解决 /autoplan 评审中发现的两个关键问题：

1. 纯 IndexedDB 方案数据分散在各巡检员手机中，管理者无法跨人查看
2. 浏览器 IndexedDB 在 iOS/Android 上会被系统主动清理，记录可能静默丢失

架构变更不影响表单逻辑——巡检员还是扫码填表，只是提交时多一个后台 POST，对使用者完全透明。

## Open Questions

1. 巡检员真的会扫码吗？还是让他们打开网页从下拉列表选设备更实际？需要去车间看一眼实际流程。
2. 巡检表里具体要填什么字段？设备状态（正常/异常/待维修）、备注、巡检人姓名、日期时间——还有别的吗？
3. 管理者想看什么样的历史视图？按设备看最近 N 次？按日期看今天所有设备？按人员看？

### Sync Endpoint（P0 — 不修系统用不了）

前端 HTML 表单提交时同步执行：
1. 写入浏览器 IndexedDB（本地缓存）
2. POST JSON 到服务器 `/api/inspections` 端点
3. 服务器用 SQLite WAL 模式存储（处理并发写入，替代 `fs.appendFile`）
4. 服务器用自己的时钟盖 `created_at` 时间戳（避免巡检员手机时间不一致）

如果 POST 失败（网络瞬断），IndexedDB 中的记录打重试标记，下次页面加载时自动 retry。

**为什么不是 fs.appendFile：** 两个巡检员同时提交时，纯文件追加在 Windows 上可能产生交错损坏的 JSON。SQLite WAL 模式天然处理并发写入，且仍然是单文件部署，不引入额外数据库服务。

### Security Hardening（P1 — 上线前必须做）

| 措施 | 说明 |
|------|------|
| XSS 防护 | 历史查询页渲染时用 `textContent` 替代 `innerHTML`，显式转义 `<` `>` `&` `"` `'` |
| CORS 限制 | sync 端点的 `Access-Control-Allow-Origin` 限制到 HTML 前端的具体 origin，不放 `*` |
| 设备列表校验 | HTML 中嵌入静态设备编号列表，巡检员从 dropdown 选设备而非自由文本输入——防止 typo 产生 phantom 设备 |
| 无认证（有意识的设计选择） | 内网信任边界足够。审计追踪由服务器端记录 IP + timestamp 实现 |

### UX Improvements（P2 — 上线后尽快补）

- 提交确认动画：绿色 ✅ + "已提交"，1 秒后清空表单（视觉反馈 + 防重复提交）
- 手动输入设备编号备用入口：QR 码花了/脏了也能填

### Deferred（P3 — TODOS）

### Test Plan（需补充）

实现时至少覆盖：

- 中文字符编码往返（UTF-8 巡检人姓名/备注）
- 浏览器兼容矩阵（车间手机可能的旧 Android WebView）

## Data Schema (Draft)

巡检记录在 IndexedDB 中存储，按以下结构：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | auto-increment | 主键，自动生成 |
| device_id | string | 设备编号（来自 URL 参数 `?device=`） |
| status | enum | 正常 / 异常 / 待维修 |
| inspector | string | 巡检人姓名 |
| note | string (optional) | 备注 |
| created_at | ISO 8601 timestamp | 提交时间，自动生成 |

IndexedDB 索引：`device_id`（按设备查）、`created_at`（按日期查）、`inspector`（按人查）。

## Edge Cases

- **重复提交**: 提交后显示"已提交"并 3 秒后清空表单——防止巡检员连续点提交产生重复记录
- **空表单提交**: 设备编号、状态、巡检人三个必填项为空时不允许提交，表单边框标红提示
- **IndexedDB 存储满**: 浏览器 IndexedDB 上限通常 >100MB，单条巡检记录 <1KB。假设每天 500 条，可用 200+ 天。满后提示导出 JSON 并清空
- **同一设备连续扫描**: 每次提交为独立记录。查询时按时间倒序显示，最近一次排最前
- **换手机/清浏览器**: 数据丢失风险。在管理页面提供"导出 JSON"按钮，管理者定期从巡检员手机浏览器导出汇总。当管理者反馈"看不到完整记录"时，触发迁移到方案 B

## Success Criteria

- 巡检员能在 30 秒内完成一次设备登记（扫码→选状态→提交）
- 历史记录能按设备编号筛选查看
- 部署到服务器上，手机浏览器能正常打开使用
- 车间实际用了一周后，没人要回纸质表格

## Distribution Plan

- **部署方式**: 把 HTML 文件放到车间的内网 Web 服务器上（或直接用 npx serve 跑起来）
- **访问方式**: 巡检员手机连 WiFi，浏览器打开 `http://<服务器IP>:端口/` 或扫码打开
- **CI/CD**: 当前阶段不需要——一个 HTML 文件，改了直接覆盖部署

## What I noticed about how you think

- 你说"通用 App 太复杂/太贵/不对口"——你很清楚自己要什么、不要什么。这种"少了才够用"的判断力在 builders 里是稀缺品。
- 三选一你直接选了最朴素的 A 方案，没有贪功能。说明你真正关心的是"车间里的人今天能不能用上"，而不是"我的系统看起来多厉害"。
- 三个前提全过——你知道这是真问题、知道扫码是正确入口、知道什么时候不需要过度工程化。一句话：你知道你的用户要什么。
